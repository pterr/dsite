1. Global architecture

The Keypl SDK is divided in two major layers:

  • The ‘Secure Element Proxy API’ which allows to manage SE readers in a generic way.

  • The ‘Calypso Processing API’ which contains at least the full sets of commands for Calypso Portable Object and Security Module, and the macro commands to manage secure Calypso transactions. This module could also be extended with features to support the Hoplink application.

Dedicated reader’s plugins have to be implemented in order to interface the SE Proxy API with the specific reader’s drivers.

2. 'Secure Element Proxy' API

2.1. Functional processing

The SE Proxy API must provide a common way to manage local or remote SE readers, providing a common interface to address NFC mobile environment or PC/SC compliant readers, or any proprietary drivers.

  • The PC/SC API provides basic functions to list and select compliant readers, to establish a connection, and to transfer APDU commands and get the corresponding responses, one by one.

  • The Smart Card IO API, used as PC/SC interface for J2SE application, provides a higher abstraction level by automatically managing 61XXh and 6CXXh warning status. In this case, the usage of the Get Response command is no longer required, ISO 7816-4 APDU command could be managed in the same way whatever the transmission protocol ISO 7816-3 T=0 or T=1, case 4 APDU commands are always processed as in contactless.

    • ‘Case 4’ commands shall always be set with ’Le’ at 00h.

    • ‘Case 2’ commands must not be set with a ‘Le’ value higher than the number of available bytes.

  • The Open Mobile API, defined to address SE in mobile environments, adds another step of abstraction with the automatic allocation of logical channels. In this case, the Select Application command is useless; a logical channel opening is requested for a specific AID, then API performs directly the selection for the first available or the specified logical channel. The class bytes of the coming APDU commands are automatically updated for the selected logical channel.

To simplify the management of logical channels with a SE, the SE Proxy API limits the establishment of only one logical channel at a time per SE.

  • A Calypso PO supports the selection of only one application at a time.

  • A CSM HSM plugin could emulate the presence of several CSM SE.

To optimize the interfacing with remote SE readers (PO or SAM):

  • The SE proxy API may manage grouped APDU commands: in order to perform operations comprised of sets of APDU requests and get the corresponding set of APDU responses through a single call. It may still be possible to emulate an APDU by APDU sequential transaction by using commands groups containing a single APDU command.

  • A single SE reader call may allow to specify at once the AID of the application to select (in order to open a specific logical channel if necessary), to transmit a group of APDU commands to play, and to define the final logical channel status after the receipt of the last APDU response of the group (in order to keep the channel open or to close it).

To support revision 1.0 and 2.4 Calypso POs in contacts T=0 mode, APDU commands’ information may declare if the ISO 7816-4 APDU is a case 4. If a case 4 response misses its data bytes, the SE plugin may send a Get Response command to retrieve the missing data.

Plugins’ rules:

  • On a request to transmit a set of APDU commands to a specific SE Application, the SE plugin or its lower driver must open a logical channel and select the targeted application (if not already selected) without expecting an explicit SelectApplication command from the SE Proxy.

  • Only one single logical channel could be open at a time with a SE Reader.

    • A smartcard reader with several physical interfaces (e.g.: several contactless antennas, or several SAM slots) ⇒ should be managed through several SE Proxy.

    • For a Secure Element supporting several logical channels at a time through a physical interface: the logical channels could be managed though different SE Proxy.

Readers’ specific settings (e.g. contactless protocol sequences polling, communication speed, …) may be managed at the plugin level. The SE proxy API provides a generic way to push or get specific reader’s parameters.

On a terminal, the SE Proxy API could be interfaced with several plugins at a time, in order to address PO and CSM, to operate both local and remote readers, supporting both PC/SC and proprietary readers.

All the classes and interfaces defining the SE Proxy Service will be contained in the package ‘cna.sdk.seproxy’.

2.2. SE Proxy - APDU command transmission (supported by all plugins)

2.2.1. final Class: SeProxyService

This final class is a singleton, the entry point of the SE Proxy Service, its instance has to be <b>called by a ticketing application<b/> in order to establish a link with a SE’s application.

  • static private Attribute: SeProxyService uniqueInstance

    • singleton’s instance of SeProxyService

  • private Attribute: Integer version

    • version number of the SE Proxy Service API

  • private Attribute: ReadersPlugin[0..*] plugins

    • the list of readers’ plugins interfaced with the SE Proxy Service

  • private Method: SeProxyService()

    • the singleton constructor

  • static Method: SeProxyService getInstance()

    • returns the singleton instance.

  • Method: Integer getVersion()

    • returns the version number of the SE Proxy Service API.

  • Method: setPlugins(ReadersPlugin[1..*] plugins)

    • add readers’ plugins to the service

  • Method: ReadersPlugin[0..*] getPlugins()

    • returns the list of interfaced reader’s plugins.

2.2.2. Interface (or abstract Class): ReadersPlugin

This object could be defined as an interface in Java, or as an abstract class in C++.

This interface (or abstract class) has to be implemented (extended) by each plugins of readers’ drivers. * Abstract method: String getName() ** returns the ‘unique’ name of the readers’ plugin.

  • Abstract method: ProxyReader[0..*] getReaders()

    • returns the list of available connected SE readers for the selected plugin.

2.2.3. Interface (or abstract Class): ProxyReader

This interface (or abstract class) has to be <b>implemented (extended) by each plugins of readers’ drivers<b/>. * Abstract method: String getName() ** returns the ‘unique’ name of the SE reader for the selected plugin.

  • Abstract method: SeResponse transmit(SeRequest seApplicationRequest) throw ChannelStateReaderException, InvalidApduReaderException, IOReaderException, TimeoutReaderException, UnexpectedReaderException

    • transmits a request to a SE application and get back the corresponding SE response

    • the usage of this method is conditioned to the presence of a SE in the selected reader, this method would fail if the SE is missing, or in case of IO communication error with the reader or the SE, if the APDU format isn’t recognize by the reader, or in case of timeout for a remote reader.

      • Reader’s plugins’ implementations have to throw the specific exceptions defined by the SDK: ChannelStateReaderException, InvalidApduReaderException, IOReaderException, TimeoutReaderException if managed by the reader, otherwise UnexpectedReaderException (for example in case of security access error…).

  • Abstract method: Boolean isSePresent()

    • returns true if a Secure Element is present in the reader

    • a useful method for “local” readers, maybe for “remote readers” the response could always be implicitly “true”.

2.2.9. Class: SeRequest

This class aggregates the elements of a request to a local or remote SE Reader, sent through a ProxyReader, in order to open a logical channel with a SE application to select, and to transfer a group of APDU commands to run.

  • Allows to open a logical channel with an application, to transfer a group of APDU command requests, and defining the final status of the logical channel (open or closed).

  • Returns the previous status of the logical channel and get a group of APDU command responses. Dedicated methods to open or close a channel are useless!

  • private Attribute: ByteArray aidToSelect[0..1]

    • AID’s bytes of the SE application to select. In case the SE application is currently not selected, a logical channel is established, and the corresponding SE application is selected by the SE reader, otherwise keep the current channel.

    • Could be missing when operating SE which don’t support the Select Application command (as it is the case for CSM).

  • private Attribute: Boolean keepChannelOpen

    • the final logical channel status:

      • if true, the SE reader keep active the logical channel of the SE application after processing the group of APDU commands.

      • if false, the SE reader will close the logical channel of the SE application after processing the group of APDU commands (i.e. after the receipt of the last APDU response).

  • private Attribute: ApduRequest[0..*] apduRequests

    • contains a group of ApduCommand to operate on the selected SE application by the SE reader.

  • Method: SeRequest(ByteArray aidToSelect[0..1], ApduRequest[0..*] apduRequests, Boolean keepChannelOpen)

    • the constructor called by a terminal in order to open a logical channel and/or to send a set of APDU commands to a SE through a ProxyReader.

    • aidToSelect could be missing to operate SE that don’t support the Select Application command (as CSM):

      • in this case, the SE Reader Plugin could avoid to do a Select Application command to open a logical channel with the default application.

      • Depending on the SE Reader Plugin, the presence of the AID could be required, as it is the case for the Android Open Mobile API plugin.

  • Method: ByteArray getAidToSelect()

    • returns the current AID set to select

  • Method: ApduRequest[0..*] getApduRequests()

    • returns the group of APDUs to be transmitted to the SE application for this instance of SeRequest.

  • Method: Boolean askKeepChannelOpen()

    • returns whether or not to keep the channel open after exchanging the APDU requests with the SE.

2.2.10. Class: ApplicationRequest

This class aggregates the elements of a request to a local or remote SE Reader, sent through a ProxyReader, in order to open a logical channel with a SE application to select, and to transfer a group of APDU commands to run.

  • Allows to open a logical channel with an application, to transfer a group of APDU command requests, and defining the final status of the logical channel (open or closed).

  • Returns the previous status of the logical channel and get a group of APDU command responses. Dedicated methods to open or close a channel are useless!

  • private Attribute: ByteArray aidToSelect[0..1]

    • AID’s bytes of the SE Application to select. In case the SE Application is currently not selected, a logical channel is established, and the corresponding SE application is selected by the SE reader, otherwise keep the current channel.

    • Could be missing when operating SE which don’t support the Select Application command (as it is the case for CSM).

  • private Attribute: Boolean keepChannelOpen

    • the final logical channel status:

      • if false, the SE reader will close the logical channel of the SE application after processing the group of APDU commands (i.e. after the receipt of the last APDU response).

  • private Attribute: ApduRequest[0..*] apduRequests

    • contains a group of ApduCommand to operate on the selected SE application by the SE reader.

  • Method: SeRequest(ByteArray aidToSelect[0..1], ApduRequest[0..*] apduRequests, Boolean keepChannelOpen)

    • the constructor called by a terminal in order to open a logical channel and/or to send a set of APDU commands to a SE through a ProxyReader.

    • aidToSelect could be missing to operate SE that don’t support the Select Application command (as CSM):

      • in this case, the SE Reader Plugin could avoid to do a Select Application command to open a logical channel with the default application.

      • Depending on the SE Reader Plugin, the presence of the AID could be required, as it is the case for the Android Open Mobile API plugin.

  • Method: ByteArray getAidToSelect()

    • returns the current AID set to select

  • Method: ApduRequest[0..*] getApduRequests()

    • returns the group of APDUs to be transmitted to the SE application for this instance of SeRequest.

  • Method: Boolean askKeepChannelOpen()

    • returns whether or not to keep the channel open after exchanging the APDU requests with the SE.

2.2.11. Class: SeResponse

This class aggregates the elements of a response from a local or remote SE Reader, received through a ProxyReader, including a group of APDU responses and the previous status of the logical channel with the targeted SE application.

  • private Attribute: Boolean channelPreviouslyOpen

    • is defined as true by the SE reader in case a logical channel was already open with the target SE application.

  • private Attribute: ApduResponse[0..1] fci

    • present if channelPreviouslyOpen was false and if a Select Application command has been processed by the SE Reader Plugin, contains the FCI response of the channel opening (either the response of a SelectApplication command, or the response of a GetData(‘FCI’) command).

  • private Attribute: ApduResponse[0..*] apduResponses

    • could contain a group of ApduResponse returned by the selected SE application on the SE reader.

  • private Attribute: ByteArray[0..1] atr

    • present if channelPreviouslyOpen is false and if the SE Reader manages the ATR (as for ISO 7816-3 contacts reader, or B’ Innovatron contactless reader), contains the ATR.

  • Method: SeResponse(Boolean[1] channelPreviouslyOpen, ApduResponse[0..1] fci, ApduResponse[0..*] apduResponses, ByteArray[0..1] atr)

    • the constructor called by a ProxyReader during the processing of the ‘transmit’ method.

  • Method: Boolean wasChannelPreviouslyOpen()

    • returns the previous state of the logical channel.

  • Method: ApduResponse[0..1] getFci()

    • returns null or the FCI response if a channel was opened.

  • Method: ApduResponse[0..*] getApduResponses()

    • returns the group of APDUs responses returned by the SE application for this instance of SeResponse.

  • Method: ByteArray[0..1] getAtr()

    • returns null or the ATR if ATR is supported by the SE Reader and if a channel was opened.

2.2.12. Class: ApduRequest

This class defines the elements of a single APDU command request: * private Attribute: ByteArray bytes an array of the bytes of an Apdu request (none structured, including the header and the dataIn of the command). * private Attribute: Boolean case4 a ‘case 4’ flag in order to explicitly specify, if it’s expected that the APDU command returns data.

  • Method: ApduRequest(ByteArray bytes, Boolean case4)

    • the constructor called by a ticketing application in order to build the APDU command requests to push to the ProxyReader.

  • Method: ByteArray getBytes()

    • returns the data of the APDU request.

  • Method: Boolean isCase4()

    • returns the case4 flag.

Support of old Calypso products not compliant with ISO 7816-4 for ‘case 4’ APDU in contacts mode:

  • Calypso Revision 2.4 Portable Objects and ‘S1Dx’ SAM replies on a case 4 APDU in contacts mode the successful status 9000h instead of 61XYh → in this case the Reader Plugin wouldn’t be able to automatically request a GetResponse command for a length XYh.

  • To recover this, case 4 APDU could be highlighted with this ‘case 4’ flag: then in case of missing data in the response with a successful status, the plugin will have to request a GetResponse for a length 00h.

2.2.13. Class: ApduResponse

This class defines the elements of a single APDU command response:

  • private Attribute: ByteArray bytes

    • an array of the bytes of an APDU response (none structured, including the dataOut field and the status of the command).

  • private Attribute: Boolean successful

    • the success result of the processed APDU command.

    • To study, if there is an interest to keep this successful attribute, maybe a processing could be to add to ApduResquest a list of “expected status”, and to stop the SeRequest processing in case of failure.

  • Method: ApduRequest(ByteArray bytes, Boolean successful)

    • the constructor called by a ProxyReader in order to build the APDU command response to push to a ticketing application.

  • Method: ByteArray getBytes()

    • returns the data of the APDU response.

  • Method: Boolean isSuccessful()

    • returns the status of the command transmission.

Apdu command transmission chaining:

  • An optional attribute could provide the list of expected SW1SW2 status: in order to stop SeRequest processing in case of unexpected status.

2.3. SE Proxy - Readers' settings (configurable & observable options for plugins)

SE Reader able to notify about its status’ change (Optional for Reader’s Plugins)

In order to notify a ticketing application in case of specific reader events (IO Error, SE Insertion or SE Removal), the SE Proxy implements the ‘Observer’ design pattern.

  • A terminal application willing to get notifications from a SpecifcReader must implement ReaderObserver. By casting the ProxyReader as ObservableReader, the application could register its SpecificObserver.

  • To issue notifications a plugin must instantiate a subclass of NotiferReader, its SpecificReader could then notify registered ReaderObserver.

2.3.1. abstract Class: ObservableReader

This abstract class could be “optionally” subclassed by the plugins of readers’ drivers. ObservableReader extends ProxyReader.

  • private Attribute: ReaderObserver[0..*] readerObservers

    • an array referencing the registered ReaderObserver of the Reader.

  • final method: void attachObserver(ReaderObserver calledBack)

    • add a ReaderObserver to the list of registered ReaderObserver for the selected ObservableReader.

  • final method: void detachObserver(ReaderObserver calledBack)

    • remove a ReaderObserver from the list of registered ReaderObserver for the selected ObservableReader.

2.3.2. abstract Class: NotifierReader

This abstract class could be “optionally” subclassed by the plugins of readers’ drivers. NotifierReader extends ObservableReader and implements ProxyReader.

  • final method: void notifyObservers(ReaderEvent event)

    • push a ReaderEvent of the selected ObservableReader to its registered ReaderObserver.

  • The methods defined by the ProxyReader interface are abstract in NotifierReader.

2.3.3. Class: ReaderEvent

This class is used to notify an event of a specific ObservableReader / NotifierReader to its registered ReaderObserver in case of IO Error, SE insertion or removal.

  • Inner Enumeration: EventType

    • defined with the elements: ‘IO_ERROR’, ‘SE_INSERTED’, & ‘SE_REMOVAL’.

  • private Attribute: NotifierReader reader

    • the reader pushing the notification.

  • private Attribute: EventType event

    • the type of the notified event.

  • Method: ReaderEvent(NotifierReader reader, EventType event)

    • the constructor

  • Method: NotifierReader getReader()

    • returns the reader notifying the event.

  • Method: EventType getEventType ()

    • returns the type of the event.

2.3.4. Interface (or abstract Class): ReaderObserver

This interface (or abstract class) could be “optionally” implemented (extended) by ticketing applications interested in receiving notification from ObservableReader.

For example, it’s useful to implement in the ticketing application of a validator in order to start (or stop) a transaction in case of SE detection (lost).

  • Abstract method: void notify(ReaderEvent event)

    • called by an ObservableReader to push a specific ReaderEvent.

2.3.5. Interface (or abstract Class): ConfigurableReader

This interface (or abstract class) could be “optionally” implemented (extended) by the plugins of readers’ drivers. ConfigurableReader extends ProxyReader.

  • Abstract method: Map<String, String> getParameters()

    • returns the configuration of the selected reader.

  • Abstract method: String setAParameter(String key, String value) throw WrongParameterKeyException, InconsistentParameterValueException, ParameterProcessingErrorException

    • could be used to set a single parameter.

    • The method returns a String, thus, it could also be used to manage specific request/response setup operations with a reader.

  • Abstract method: void setParameters(Map<String, String> settings) throw WrongParameterKeyException, InconsistentParameterValueException, ParameterProcessingErrorException

    • allows to define some or all proprietary settings for a plugin (contactless protocols polling sequence, baud rate, … etc.) by setting a key-value Map of parameters.

  • Abstract method: Map<String, String> getParameters()

    • returns the configuration of the selected reader.

3. 'Generic Command' API

The 'Commands' module is the base to define the APDU commands’ sets for a specific Secure Element application. All these commands could be operated independently in standalone.

To manage SE commands, the library requires:

  • functions to build APDU requests in order to generate the arrays of bytes to send to a SE in a group of APDU requests (SeRequestElement),

  • and functions to parse the arrays of bytes replied by the SE in a group of APDU responses (SeResponseElement).

3.1. Interface: CommandsTable

This interface allows to define a register of Commands’ Sets including association between Commands’ Builders and Responses’ Parsers. It provides the generic getters to retrieve: the name & the instruction byte of a command, and the corresponding ApduCommandBuilder & ApduResponseParser classes.

  • Method: String getName()

    • returns the command name.

  • Method: Byte getInstructionByte()

    • returns the value of the INS byte.

  • Method: Class getCommandBuilderClass()

    • returns the Command Builder class.

  • Method: Class getResponseParserClass()

    • returns the Response Parser class.

3.2. abstract Class: ApduCommandBuilder

This abstract class has to be extended by all SE command builder classes, it provides the generic getters to retrieve: the name of the command, the built ApduRequest, the corresponding ApduResponseParser class.

  • private Attribute: CommandsTable commandReference

    • the reference of the command in the matrix array enumeration, in order to get the name and the response parser class of the command.

  • private Attribute: ApduRequest request

    • the byte array APDU request.

    • For ‘case 4’ APDU command, the ‘Le’ field shall be present with the value 00h.

  • abstract Method: ApduCommandBuilder(CommandsTable reference, ApduRequest request)

    • the generic abstract constructor to build an APDU request with a command reference and a byte array.

  • final Method: String getName()

    • returns the name of the APDU command from the CommandsTable information.

  • final Method: ApduRequest getApduRequest()

    • returns the APDU request.

  • final Method: Class getApduResponseParserClass()

    • returns the corresponding ApduResponseParser class of the APDU command from the CommandsTable information.

3.3. abstract Class: ApduResponseParser

This abstract class has to be extended by all SE response parser classes, it provides the generic getters to manage response’s status.

  • private Attribute: ApduUResponse response

    • the byte array APDU response.

  • protected inner Class (or struct) StatusProperties

    • protected Attribute: Boolean successful

    • protected Attribute: String information

  • protected Attribute: Map<Short, StatusProperties>: statusTable

    • A map with the double byte of a status as key, and the successful property and ASCII text information as data.

  • abstract Method: ApduResponseParser(ApduResponse responseToParse, Map<Short, StatusProperties>)

    • the generic abstract constructor to build a parser of the APDU response, it sets the ApduResponse and the Map of StatusProperties pushed by the sub-classes.

  • final Method: ApduResponse getApduResponse ()

    • returns the ApduResponse instance.

  • final Method: Short (or unsigned short) getStatusCode)

    • Returns SW1-SW2 bytes from the ApduResponse.

  • final Method: Boolean isSuccessful()

    • Returns if the status is successful from the statusTable according to the current status code.

  • final Method: String getStatusInformation()

    • Returns the ASCII message from the statusTable for the current status code.
